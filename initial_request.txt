Recommended module pattern
Design your code so core logic is environment-agnostic:
spark_session.py: creates/configures SparkSession
jobs/my_job.py: orchestration (read -> transform -> write)
transforms/*.py: pure DataFrame transforms
io/*.py: data source/sink helpers
Key idea: write functions like transform_orders(df: DataFrame) -> DataFrame so they run identically local + Databricks.
Local run strategy
Use a local session config like:
master("local[*]")
low shuffle partitions (spark.sql.shuffle.partitions=4) for dev speed
temp local paths for I/O
Then run via Poetry (poetry run ...) to stay in your project env.
Databricks run strategy
Two good options:
Notebook-first: %pip install your package wheel, then call your module
Job-first: build wheel in CI, attach/install on cluster, run Python task/notebook
If your local dev must closely match DBR behavior, use Databricks Connect for remote execution from local IDE (best fidelity), while still keeping fast local unit tests with local[*].
Testing strategy (important)
Unit tests locally: small in-memory DataFrames, deterministic assertions
Contract tests locally: schemas/nullability/edge cases
Integration tests in Databricks: file formats, permissions, cluster/runtime behavior
This gives fast feedback without over-investing in full local emulation.
